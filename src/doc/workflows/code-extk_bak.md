# 任务目标

分析当前项目中（基于 langchain4j 框架开发）的新增或有改动的方法，从中提取相关的 AI 和框架概念（如多模态、系统提示词等）。然后用大白话解释这些概念，提供它们在 langchain4j 框架中的具体代码使用示例（包括入参和出参等），并以增量补充的方式记录到 Markdown 历史总结文件中。同时，如果代码方法缺少相关注解，需主动补充。
并且，每次执行完毕后，需要生成一份**变更总结日志**，记录本次分析对比上次新增或变更了哪些知识点内容，按天归档。

**注意：**
1. 在任何分析中，**始终完全忽略**位于项目 `src/test` 目录下的所有测试用例代码。只关注 `src/main` 等正式业务代码中的改动。
2. **完全忽略**所有常量（Constants）和枚举（Enums）类型的代码变更，不对它们进行任何分析。
3. 核心知识总文件（必须更名为 `learning_record.md`）存放在 `src/doc` 目录下。
4. **每日变更日志**必须存放在 `src/doc/extract_log` 目录下。如果该目录不存在，必须自动创建。
5. **如果没有发现任何符合调件的代码调整或未提取到任何新知识点，则直接结束，什么记录也不做，也不要创建或修改总结与日志文件。**
6. **绝对不要执行 `git status` 或任何其他 git 命令**来寻找变更代码。请基于用户当前的上下文（如 Active Document）或直接搜索、对比现有代码和历史记录来进行分析。

# 执行步骤

1. **基于轻量级元数据留痕的全局分析变更范围（极速增量分析）**
   - 检查并确保项目 `src/doc/` 目录下存在历史总结文件 `learning_record.md` 和日志目录 `extract_log/`。
   - **核心扫描范围**：每次执行必须全局扫描 `src/main/java` 目录下所有的业务代码文件，不能仅仅局限于当前 Active Document。
   - **执行跨平台快照对比脚本**：首先在终端中执行命令 `node src/doc/extk_traces/diff_trace.js`，该脚本会瞬时分析全项目 Java 文件的 MD5 哈希，并直接在终端输出具体哪些文件发生了变动（或者输出 `NO_CHANGES_DETECTED`）。
   - **增量比对策略**：
     - **如果脚本输出 `NO_CHANGES_DETECTED`**：说明没有任何文件发生变动，直接结束本次执行流程，不留任何记录。
     - **如果输出了新增或变动的文件列表**：直接根据脚本提供的文件路径读取这些发生变动的文件，去和 `learning_record.md` 中的总结进行知识点比对并提取全新的 AI 概念细节。
   - **更新防串改哈希底库**：在所有的知识点提取与重写动作全部完毕后，**必须及时在终端执行** `node src/doc/extk_traces/gen_trace.js`，将本次最新的正确状态全部写入覆盖到 `src/doc/extk_traces/extk_trace_state.json` 文件中，供下一次光速比对。

2. **自动补充代码注释（知识点与作用）**
   - 在总结某个方法时，**必须检查代码中该方法上方是否有 JavaDoc 注释**。
   - 如果方法已有注释，需理解该注释是否准确描述了方法的作用和相关的AI概念。如果有误，进行修正；如果准确，则**坚决不要**修改它。如果没有注释，则补充生成新的 JavaDoc。
   - 注释内容写明：1. **该方法的具体作用**； 2. **该方法涉及到的底层 AI 概念或 langchain4j 知识点**。

3. **概念提取、白话文解释与框架实现描述**
   - 从新增/修改的方法中提取核心 AI 概念或框架特性。
   - **核心要求**：对于大的知识点，必须首先明确提取并生成该“概念”本身的正式定义或介绍。**注意：在阐述完概念的通用定义后，必须在对应的知识点下，简要地描述一下该概念在 langchain4j 框架中是如何具体实现的（框架中的实现方式或架构思路）**。
   - **分析注解中的“注意”事项**：检查代码的方法注解或类注解中是否包含“**注意**”等警示性说明。如果存在，必须对其做深入剖析，理解其背后的使用限制、潜在风险或架构设计考量，并将这些注意点提炼、归纳到对应的核心知识点中一并输出。
   - **然后**再使用生动形象的大白话或者生活中的比喻，为非技术人员解释这个概念。

4. **提供框架使用示例与代码**
   - 讲述该概念在 **langchain4j** 框架中怎么具体使用，将框架特有的内容和实现统一放到本部分展示。统一固定使用 `### 框架使用示例` 作为子标题，**绝对不要**出现类似于“框架使用实例”和“框架使用实例与不同实现方式”这样额外的小分类或并存情况。
   - **当存在多种实现方式时的处理基准**：如果发现该知识点有不同的实现方式（例如：原生的手动实现方式 vs 注解式或高级特性的实现方式），**不要把多种方式都列出，也不要去做大篇幅的优缺点对比分析。请直接以第二种方式（即基于框架的高级/最新实现方式）为基准，只展示该基准方式的核心代码示例即可**。
   - **核心要求**：在举例说明时，**不能仅局限于当前代码中使用的类型。必须系统性地补充说明该概念在 langchain4j 中还支持哪些其他相关类型。** 例如，对于“多模态”知识点，不能只提代码里出现的 `TextContent` 和 `ImageContent`，必须明确指出它还能支持 `AudioContent`、`VideoContent` 等等，以提供完整的知识面。
   - 给出对应的代码示例，加上注释明确说明入参、出参和处理逻辑。

5. **增量生成 Markdown 主总结**
   - 将新提取的知识点、白话文解释和代码，**新增补充** 到 `src/doc/learning_record.md`。
   - **格式要求（重要）**：在追加新的知识点总结或代码示例时，**务必仔细检查并确保前文以及当前生成的 Markdown 代码块（```）已被正确闭合**。绝对不要把新章节的标题（如 `## 5. xxx`）或正文错误地生成在上一个例子的代码块内部。

6. **📝 记录每日增量日志**
   - 执行完上述步骤后，梳理本次执行“新增了什么知识点”、“修改/扩展了什么老知识点”。
   - 在 `src/doc/extract_log` 目录下，以**当天日期**命名文件（例如：`2024-03-03.md`）。
   - 如果当天对应的日志文件已存在，则在文件末尾**追加**本次执行的变更记录清单，确保一天的所有总结执行日志都在同一个 Markdown 文件中合并记录。
   - **日志内容示例：**
     - 【时间】：`13:45 提取执行`
     - 【新增总结】：首次分析了 `SystemMessage` 概念（系统提示词），通过大白话解释并补充了入参出参代码示例。
     - 【代码变更】：为 `FirstSightChatAgent.chatWithSystemMessage` 补充了业务与 AI 概念 JavaDoc 注释。
